# 173. Amazon ECS Task Definitions - Hands On

## 🎯 Giới thiệu
- Bài này đi qua toàn bộ các tùy chọn khi tạo **ECS Task Definition** trong console.
- Trọng tâm là cách cấu hình **infrastructure requirements**, **role**, **container settings**, **logging**, **storage**, và **monitoring**.
- Đây là phần rất quan trọng cho kỳ thi vì nhiều chi tiết dễ nhầm giữa **Fargate** và **EC2**, cũng như giữa **task role** và **task execution role**.

## 1. Infrastructure Requirements
- Khi tạo task definition, bạn chọn **family name** như `wordpress`.
- Task có thể chạy trên:
  - **Fargate**
  - **Amazon EC2 instances**
  - Hoặc cả hai
- Nếu dùng **Fargate**:
  - Phải chọn **CPU** và **memory** theo các mức tương thích với Fargate.
  - **Network mode** phải là **AWS VPC**
- Nếu chỉ dùng **EC2**:
  - Có thể nhập giá trị **CPU** và **memory** linh hoạt hơn.
  - Có thể chọn **network mode** nâng cao hơn

## 2. Task Role và Task Execution Role
- **Task role**:
  - Rất quan trọng trong exam
  - Cho phép container gọi **AWS API**
  - Là IAM role mà ECS task sử dụng
- **Task execution role**:
  - Là **IAM role** dành cho **container agent**
  - Dùng để thực hiện **AWS API requests** thay mặt bạn
  - Đây là role mang tính “chuẩn ECS” hơn cho phần thực thi

## 3. Container Configuration
- Một task definition có thể có **nhiều containers**
- Ít nhất một container phải được đánh dấu là **essential container**
- Nếu một **essential container** bị lỗi và bị kill:
  - Cả task sẽ bị dừng
- Nếu container **không essential**:
  - Container đó có thể dừng mà task vẫn tiếp tục chạy
- Container image:
  - Có thể lấy từ **public repository**
  - Hoặc **private repository**
  - Nếu private thì cần xác thực bằng **Secrets Manager ARN**
- **Port mapping**:
  - Khai báo ports, protocols, port name, type of protocol
  - Hỗ trợ `HTTP`, `HTTP2`, `GRPC`, hoặc `none`
- Có thể thêm nhiều **port mappings** nếu application expose nhiều port
- Có thể giới hạn tài nguyên:
  - **VCPU**
  - **memory**
  - **hard limit**
  - **soft limit**
- **Environment variables**:
  - Set trực tiếp theo dạng key-value
  - Hoặc lấy từ **Secrets Manager**
  - Hoặc từ **SSM Parameter Store**
  - Hoặc từ file host trên **Amazon S3**
- **HealthChecks**:
  - Dùng để kiểm tra container còn healthy hay không
- **Timeouts**:
  - **start timeout**: start quá chậm thì bị kill
  - **stop timeout**: bị kill trước khi kết thúc đúng cách
- Có thể cấu hình thêm:
  - **Docker configuration**
  - **Docker labels**
  - resource-specific settings

## 4. Storage, Logging và Monitoring
- **Storage**:
  - Có thể mount thêm storage cho containers
  - Dùng **bind mount**
  - Hoặc mount **EFS file system**
  - Có thể mount từ:
    - volume được định nghĩa
    - container khác
- **Logging**:
  - Có thể gửi logs tới:
    - **CloudWatch**
    - **Splunk**
    - **Firehose**
    - **Kinesis Stream**
    - **OpenSearch**
    - **S3**
  - Có thể dùng **AWS FireLens** hoặc gửi native trực tiếp tới **CloudWatch**
  - Nếu dùng CloudWatch, cần chỉ rõ:
    - **log group**
    - **log region**
    - **stream prefix**
    - có tạo group hay không
- **Monitoring**:
  - Có thể bật **trace collection** để gửi dữ liệu vào **AWS X-Ray**
  - Thông qua sidecar **AWS Distro for Open Telemetry**
  - CPU và memory sẽ tự điều chỉnh để tính cho sidecar này
  - Có thể collect metrics và gửi vào:
    - **CloudWatch**
    - **Managed Service for Prometheus**

## 📊 Bảng tóm tắt
| Tiêu chí | Mô tả |
|----------|------|
| Compute | **Fargate** hoặc **Amazon EC2 instances** |
| Fargate constraints | CPU/memory phải theo mức tương thích, network mode là **AWS VPC** |
| EC2 flexibility | CPU/memory nhập linh hoạt hơn, network mode nâng cao hơn |
| IAM roles | **Task role** cho ECS task, **task execution role** cho container agent |
| Containers | Có thể có nhiều container, ít nhất một **essential container** |
| Private image access | Dùng **Secrets Manager ARN** để xác thực |
| Networking | **Port mapping** cho nhiều port và protocol |
| Environment variables | Nhập tay, lấy từ **Secrets Manager**, **SSM Parameter Store**, hoặc file trên **S3** |
| Logging | Gửi tới **CloudWatch**, **Splunk**, **Firehose**, **Kinesis Stream**, **OpenSearch**, hoặc **S3** |
| Storage | **bind mount**, **EFS**, hoặc mount giữa containers |
| Monitoring | Trace tới **AWS X-Ray**, metrics tới **CloudWatch** hoặc **Managed Service for Prometheus** |

## 💡 Mẹo ghi nhớ cho kỳ thi AWS
- Nhớ ngay: **Fargate = AWS VPC network mode bắt buộc**
- Phân biệt rõ:
  - **Task role** = quyền cho application/container gọi AWS API
  - **Task execution role** = quyền cho container agent khi ECS thực thi
- **Essential container** chết thì **task chết theo**
- Env vars có thể đến từ:
  - trực tiếp
  - **Secrets Manager**
  - **SSM Parameter Store**
  - file trên **S3**
- Logs có thể đi tới nhiều đích, nhưng **CloudWatch** là đường native được nhắc tới rõ nhất
- Storage có thể gắn từ **EFS** hoặc từ volume/container khác
- Trace collection dùng sidecar **AWS Distro for Open Telemetry** để đẩy vào **AWS X-Ray**

## ✅ Kết luận
- **ECS Task Definition** là nơi mô tả đầy đủ cách task chạy: compute, role, container, storage, logging, và monitoring.
- Điểm thi thường xoay quanh sự khác nhau giữa **Fargate vs EC2**, **task role vs task execution role**, và ý nghĩa của **essential container**.
- Nắm đúng các cấu hình này giúp bạn hiểu cách ECS triển khai và vận hành task một cách chính xác.
