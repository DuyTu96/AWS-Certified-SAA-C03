# 83. Lambda@Edge and CloudFront Functions

## 🎯 Giới thiệu
- CloudFront không chỉ là CDN, mà còn cho phép chạy **custom code at the edge** để xử lý logic sớm, gần người dùng nhất.
- Mục tiêu chính:
  - giảm **latency**
  - kiểm tra logic sớm
  - chuẩn hóa **cache key**
  - thay đổi **request/response** trước khi đi tiếp
- Có 2 loại edge functions trong CloudFront:
  - **CloudFront Functions**
  - **Lambda@Edge**
- Cả hai đều là **serverless**, không cần quản lý server và được deploy globally.

## 1. Vị trí chạy và luồng hoạt động
- **CloudFront Functions** chạy ở **edge locations**.
- **Lambda@Edge** chạy ở **regional edge cache**.
- CloudFront Functions chỉ tác động lên:
  - **viewer request**
  - **viewer response**
- Lambda@Edge có thể tác động lên:
  - **viewer request**
  - **origin request**
  - **origin response**
  - **viewer response**

```mermaid
flowchart LR
    A[Client] --> B[Edge Location]
    B --> C[Regional Edge Cache]
    C --> D[Origin]

    A -. viewer request / viewer response .-> B
    C -. origin request / origin response .-> D

    F[CloudFront Functions] --> B
    L[Lambda@Edge] --> C
```

## 2. So sánh CloudFront Functions và Lambda@Edge
### CloudFront Functions
- Dùng **JavaScript**.
- Là **native feature** của CloudFront.
- Quản lý code trực tiếp trong CloudFront.
- Chỉ dùng cho thay đổi rất nhanh, rất nhẹ:
  - đọc/chỉnh sửa header
  - normalize request
  - rewrite/redirect
  - request filtering
  - authentication/authorization đơn giản
- Không gọi được **external services**.
- Không có **cache behavior** riêng.
- Không có **network access**, **file system access**, và không access được **request body**.
- Rất nhanh:
  - **sub-millisecond startup**
  - **1 millisecond execution time**
  - có thể xử lý **millions of requests per second**
- **Process-based isolation**
- Package rất nhỏ: **10 KB**
- Memory thấp: **2 MB**
- Có **free tier**
- Giá thấp hơn Lambda@Edge, khoảng **1/6**

### Lambda@Edge
- Dùng **NodeJS** hoặc **Python**.
- Phù hợp khi cần logic mạnh hơn và linh hoạt hơn.
- Có thể:
  - chạy lâu hơn
  - gọi **third party API**
  - dùng **network access**
  - dùng **file system access**
  - access **request body**
  - dùng **SDK** để truy cập dịch vụ AWS khác
- **VM-based isolation**
- Execution time:
  - **5 seconds** cho viewer triggers
  - **30 seconds** cho origin triggers
- Memory lớn hơn:
  - **128 MB** hoặc **10 GB**
- Không có free tier
- Tính phí theo **request** và **duration**
- Khi deploy, phải author function ở **US East 1**, sau đó CloudFront replicate tới các regional edge caches.

### Bảng so sánh nhanh
| Tiêu chí | CloudFront Functions | Lambda@Edge |
|----------|----------------------|-------------|
| Runtime | JavaScript | NodeJS, Python |
| Vị trí chạy | Edge locations | Regional edge cache |
| Trigger | Viewer request, Viewer response | Viewer request, Origin request, Origin response, Viewer response |
| Isolation | Process-based | VM-based |
| Thời gian chạy | 1 ms | 5s / 30s |
| Memory | 2 MB | 128 MB hoặc 10 GB |
| Network access | Không | Có |
| File system access | Không | Có |
| Request body | Không | Có |
| External services | Không | Có |
| Giá | Free tier, rẻ hơn | Không có free tier |

## 3. Use cases chính
### CloudFront Functions phù hợp khi:
- **Cache key normalization**
  - thay đổi headers, cookies, query string, URL để tạo cache key tối ưu
- **HTTP header manipulation**
  - insert, modify, delete headers
- **Rewrite / redirect**
- **Request filtering**
- **Authentication / authorization at the edge**
  - validate token như **JWT**
  - chặn request sớm trước khi về origin
- **Bot mitigation**
- **A/B testing**
- **Xử lý rất nhanh, đơn giản, độ trễ thấp**

### Lambda@Edge phù hợp khi:
- Cần **longer execution time**
- Cần gọi **third-party libraries**
- Cần **AWS SDK** để truy cập service khác
- Cần **network access**
- Cần **file system access**
- Cần access **HTTP request body**
- Cần logic phức tạp hơn CloudFront Functions

### Ví dụ trong transcript
- **AuthN/AuthZ**:
  - CloudFront Functions có thể kiểm tra **JWT** trong header tại edge location
  - nếu sai thì trả lỗi ngay, không gửi tiếp về origin
  - Lambda@Edge có thể làm logic kiểm tra phức tạp hơn và gọi API bên thứ ba
- **User-agent based content**:
  - desktop và mobile cùng request ảnh cat
  - CloudFront Function kiểm tra **User-Agent**
  - mobile được redirect sang ảnh độ phân giải thấp hơn
- **Global application**:
  - website từ **S3**
  - dynamic API request đi qua CloudFront
  - Lambda@Edge có thể query **DynamoDB** để tạo mô hình API global, serverless

## 📊 Bảng tóm tắt
| Tiêu chí | Mô tả |
|----------|------|
| Mục tiêu | Chạy custom code gần user để giảm latency và xử lý request/response sớm |
| CloudFront Functions | Nhẹ, nhanh, chỉ ở edge location, phù hợp transform đơn giản |
| Lambda@Edge | Mạnh hơn, chạy ở regional edge cache, phù hợp logic phức tạp |
| Trigger quan trọng | CloudFront Functions: viewer request/response; Lambda@Edge: viewer + origin request/response |
| Deploy | Lambda@Edge phải author ở **US East 1** rồi CloudFront replicate |
| Khi chọn CloudFront Functions | Cần cache key normalization, header changes, rewrite/redirect, auth đơn giản |
| Khi chọn Lambda@Edge | Cần body, network, file system, third-party API, thời gian chạy dài hơn |

## 💡 Mẹo ghi nhớ cho kỳ thi AWS
- **CloudFront Functions = “nhanh, nhẹ, ở edge”**
  - nhớ 3 ý: **viewer request/response**, **1 ms**, **không network**
- **Lambda@Edge = “mạnh hơn, ở regional edge cache”**
  - nhớ 4 ý: **origin request/response**, **5s/30s**, **body access**, **network/API**
- Nếu câu hỏi nói về:
  - **cache key normalization**
  - **header manipulation**
  - **simple auth at the edge**
  - **very low latency**
  → nghĩ ngay đến **CloudFront Functions**
- Nếu câu hỏi nói về:
  - **third-party API calls**
  - **request body**
  - **file system**
  - **longer runtime**
  → nghĩ ngay đến **Lambda@Edge**
- Câu thi thường xoay quanh việc chọn đúng công cụ theo nhu cầu:
  - **simple and fast** → CloudFront Functions
  - **complex and flexible** → Lambda@Edge

## ✅ Kết luận
- CloudFront hỗ trợ chạy custom code ở edge để xử lý request/response gần người dùng.
- **CloudFront Functions** phù hợp cho xử lý rất nhanh, đơn giản, tối ưu cache và header.
- **Lambda@Edge** phù hợp cho logic phức tạp hơn, cần runtime dài hơn, access network, body, hoặc dịch vụ bên ngoài.
- Điểm mấu chốt khi ôn thi là chọn đúng công cụ dựa trên **vị trí chạy**, **loại trigger**, và **mức độ phức tạp của logic**.
