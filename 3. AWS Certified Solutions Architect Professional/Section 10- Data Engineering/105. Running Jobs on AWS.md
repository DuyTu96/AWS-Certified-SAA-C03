# 105. Running Jobs on AWS

## 🎯 Giới thiệu
Bài này nói về các cách chạy **jobs** trên AWS, từ **CRON jobs**, **serverless scheduled jobs**, **reactive workflows** cho tới **batch / container / big data workloads**.  
Mục tiêu chính là chọn đúng dịch vụ theo tính chất công việc: ngắn hạn, dài hạn, phản ứng theo sự kiện hay xử lý dữ liệu lớn.

## 1. EC2 + CRON jobs ⏱️
- Cách đầu tiên là dùng **EC2 instance** để chạy các **long running jobs** và đặt **CRON jobs** trên đó.
- Cách này phù hợp nếu bạn đã quen với CRON.
- Tuy nhiên, đây **không phải** là chiến lược tốt về kiến trúc vì:
  - Không **highly available**
  - Không **scalable**
  - Nếu EC2 instance bị down thì job sẽ gặp vấn đề
- Kết luận của bài giảng: đây là cách mà một **Solutions Architect** nên tránh chọn.

## 2. EventBridge + Lambda / Reactive workflow ⚡
- Cách thứ hai là dùng **Amazon EventBridge** kết hợp **Lambda**.
- Đây là mô hình **serverless CRON job**:
  - **EventBridge rule** có thể trigger Lambda theo lịch, ví dụ **every hour**
  - Lambda chạy hoàn toàn serverless
- Điểm cần nhớ:
  - Bị giới hạn bởi **Lambda runtime** và **timeout**
- Ngoài scheduling, còn có mô hình **reactive workflow**:
  - Khi có sự kiện xảy ra trong AWS, Lambda được invoke ngay
  - **EventBridge** có thể nhận sự kiện từ nhiều AWS services
  - **S3** phát sinh event khi có object mới
  - **API Gateway** có thể invoke Lambda
  - **SQS** và **SNS** cũng có thể là nguồn kích hoạt Lambda
- Ý chính: hệ thống phản ứng ngay khi có event trong infrastructure.

## 3. Batch, Fargate, EMR 🧱
- Với **longer running jobs** hoặc **full-fledged containers**, có thể dùng:
  - **EventBridge + Batch**
- Ví dụ:
  - Chạy **batch job** mỗi **3 hours**
  - Dùng **time-based event rule** trong EventBridge để trigger Batch
- **Batch** phù hợp cho **long running workloads**
- **Fargate** cũng có thể dùng làm target của EventBridge để:
  - Chạy một container nhanh chóng
  - Theo kiểu “bare bones” hơn Batch
- **Batch** được mô tả là có nhiều khả năng hơn và khả năng scale tốt hơn so với Fargate trong ngữ cảnh chạy batch job
- Nếu cần chạy **very long jobs** hoặc **big data jobs**:
  - Dùng **EMR (Elastic MapReduce)**
  - Đặc biệt hữu ích khi cần **step execution**, **cluster**, và xử lý **big data workloads**

## 📊 Bảng tóm tắt
| Tiêu chí | Mô tả |
|----------|------|
| EC2 + CRON | Chạy job trên EC2, dễ hiểu nhưng không HA và không scalable |
| EventBridge + Lambda | Serverless CRON job, phù hợp job ngắn và trigger theo lịch |
| Reactive workflow | Lambda được invoke bởi event từ AWS services như S3, API Gateway, SQS, SNS |
| Batch | Phù hợp long running workloads và batch job theo lịch |
| Fargate | Target của EventBridge để chạy container nhanh chóng |
| EMR | Phù hợp very long jobs và big data workloads, có cluster và step execution |

## 💡 Mẹo ghi nhớ cho kỳ thi AWS
- **EC2 + CRON**: nhớ ngay vấn đề là **không HA, không scalable**.
- **EventBridge + Lambda**: là đáp án quen thuộc cho **scheduled serverless job**.
- **Reactive event-driven**: khi đề bài nói “có event là chạy”, nghĩ tới **Lambda** với nguồn từ **S3, API Gateway, SQS, SNS, EventBridge**.
- **Batch**: nhớ cho **long running workloads** và **batch jobs**.
- **Fargate**: dùng khi muốn chạy **container nhanh** từ EventBridge.
- **EMR**: dành cho **big data** và job rất dài.

## ✅ Kết luận
Bài giảng đưa ra 3 hướng chính để chạy jobs trên AWS:
- **EC2 + CRON**: đơn giản nhưng kém về HA và scalability
- **EventBridge + Lambda**: serverless, linh hoạt, phù hợp scheduled hoặc reactive workflow
- **Batch / Fargate / EMR**: dành cho job dài, container, hoặc big data workload
