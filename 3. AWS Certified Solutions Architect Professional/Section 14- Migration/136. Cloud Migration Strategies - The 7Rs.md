# 136. Cloud Migration Strategies - The 7Rs

## 🎯 Giới thiệu
- AWS có 7 chiến lược cloud migration thường được gọi là **The 7Rs**.
- Mục tiêu là chọn đúng hướng di chuyển workload lên cloud tùy theo:
  - mức độ cần thay đổi,
  - chi phí,
  - thời gian triển khai,
  - bảo mật,
  - compliance,
  - performance,
  - dependency.
- Đây là chủ đề hay xuất hiện trong exam khi hỏi: **nên dùng strategy nào trong từng tình huống**.

## 1. Nhóm không thay đổi nhiều: **Retire, Retain, Relocate**
- **Retire**
  - Tắt các service không còn cần dùng.
  - Giúp giảm **attack surface** và tiết kiệm chi phí.
  - Tập trung nguồn lực vào các resource cần duy trì.
- **Retain**
  - Không migrate lên cloud.
  - Vẫn giữ on-premises vì:
    - security,
    - data compliance,
    - performance,
    - unresolved dependencies,
    - không có business value,
    - hoặc quá phức tạp để chuyển.
- **Relocate**
  - Di chuyển app từ on-premises sang phiên bản cloud tương ứng.
  - Cũng có thể là chuyển **EC2 instances** sang:
    - VPC khác,
    - account khác,
    - AWS Region khác.
  - Ví dụ: vẫn dùng VMware nhưng chuyển sang **VMware Cloud on AWS**.

## 2. Nhóm chuyển lên cloud theo kiểu lift-and-shift: **Rehost, Replatform, Repurchase**
- **Rehost** = **lift and shift**
  - Migrate application, database, data, server lên AWS mà không tối ưu lại kiến trúc.
  - Có thể chuyển từ physical, virtual, server, hoặc cloud khác sang AWS.
  - App giữ nguyên, chỉ “đưa lên cloud”.
  - Có thể tiết kiệm chi phí nhờ cloud resources.
  - Ví dụ dịch vụ hỗ trợ: **AWS Application Migration Service**.
- **Replatform** = **lift and reshape**
  - Không đổi core architecture của application.
  - Nhưng chuyển sang managed service hoặc serverless service để có lợi ích cloud.
  - Ví dụ:
    - database chuyển sang **RDS**,
    - application chuyển sang **Elastic Beanstalk**.
  - Lợi ích của RDS được nhắc đến:
    - backups,
    - resiliency,
    - high availability.
- **Repurchase** = **drop and shop**
  - Chuyển sang một sản phẩm khác khi migrate lên cloud.
  - Thường là chuyển sang **SaaS**.
  - Ví dụ:
    - CRM lên **salesforce.com**,
    - HR lên **Workday**,
    - CMS lên **Drupal**.
  - Triển khai nhanh nhưng chi phí ngắn hạn có thể cao.

## 3. Nhóm tái kiến trúc: **Refactor / Re-architect**
- **Refactor and re-architect**
  - Thiết kế lại application theo hướng **Cloud Native**.
  - Mục tiêu là cải thiện:
    - scalability,
    - performance,
    - security,
    - agility.
  - Có thể:
    - tách monolithic app thành **microservices**,
    - chuyển sang **serverless architecture**,
    - dùng **Amazon S3** để lưu dữ liệu, v.v.
  - Đây là strategy tốn effort nhất nhưng thường đem lại payoff lớn nhất.

## 📊 Bảng tóm tắt
| Tiêu chí | Mô tả |
|----------|------|
| **Retire** | Tắt resource không còn cần dùng |
| **Retain** | Giữ nguyên on-premises, không migrate |
| **Relocate** | Di chuyển app/EC2 sang môi trường AWS khác mà không đổi nhiều |
| **Rehost** | Lift and shift lên AWS, ít hoặc không thay đổi kiến trúc |
| **Replatform** | Chuyển sang managed/service tối ưu hơn nhưng không đổi core architecture |
| **Repurchase** | Chuyển sang sản phẩm khác, thường là SaaS |
| **Refactor** | Thiết kế lại ứng dụng theo Cloud Native |
| **Mức effort** | Tăng dần từ Retire/Retain đến Refactor |
| **Mức thay đổi** | Cao nhất ở Refactor, thấp nhất ở Retire/Retain |
| **Khi thi AWS** | Nhận diện strategy dựa trên mức độ thay đổi và mục tiêu migration |

## 💡 Mẹo ghi nhớ cho kỳ thi AWS
- Nhớ theo thứ tự: **Retire - Retain - Relocate - Rehost - Replatform - Repurchase - Refactor**.
- Mẹo phân biệt nhanh:
  - **Retire**: bỏ luôn.
  - **Retain**: giữ nguyên.
  - **Relocate**: chuyển chỗ.
  - **Rehost**: đưa nguyên lên cloud.
  - **Replatform**: nâng cấp nhẹ sang managed service.
  - **Repurchase**: mua/đổi sang SaaS.
  - **Refactor**: viết lại theo cloud-native.
- Khi đề bài nói:
  - “không đổi kiến trúc” → nghĩ đến **Rehost** hoặc **Replatform**.
  - “chuyển sang SaaS” → **Repurchase**.
  - “thiết kế lại để tận dụng cloud” → **Refactor**.
  - “không cần migrate nữa” → **Retire**.
  - “vẫn giữ on-premises” → **Retain**.

## ✅ Kết luận
- **The 7Rs** là khung tư duy quan trọng để chọn chiến lược migration phù hợp.
- Điểm mấu chốt là xác định:
  - workload nào nên bỏ,
  - workload nào giữ nguyên,
  - workload nào chuyển nguyên,
  - workload nào tối ưu nhẹ,
  - workload nào cần tái kiến trúc sâu.
- Trong exam, hãy luôn gắn mỗi **R** với mức độ thay đổi và mục tiêu kinh doanh tương ứng.
